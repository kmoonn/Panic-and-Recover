# 索引

一种用于快速查询和检索数据的数据结构，其本质可以看成是一种排序好的数据结构。

索引的作用就相当于书的目录

B+ 树作为索引结构

## 优点、缺点

索引优点是查询快、唯一约束、加速排序分组；
缺点是降低写入性能、占用空间、设计不当会失效。

- 优点

提高查询速度、减少扫描行数与磁盘 I/O

唯一索引保证数据唯一性

加快 ORDER BY、GROUP BY，利用索引有序性避免额外排序

- 缺点

降低增删改效率：DML 操作需要同步维护索引

占用额外磁盘空间

设计不当会失效或被误用，反而影响性能

## 不建议使用的场景

数据量太小全表扫描比索引更快，因为索引本身有开销。

查询结果集占比过大如返回数据超过表的 20%~30%，优化器会选择全表扫描，避免大量**回表**（通过二级索引查到主键后，再去主键索引查完整数据）**随机 I/O**（索引查到的数据行在磁盘上不连续、散乱分布
磁头要到处跳着找数据）。

统计信息过时 / 索引维护不当优化器选错执行计划，导致索引失效或效率低下。

## 为什么快

核心原因是它大大减少了磁盘 I/O 的次数

索引是排好序的数据结构不用全表扫描，像书的目录一样直接定位数据。

MySQL 使用 B+ 树，结构 “矮胖”千万级数据的索引树通常只有 3～4 层。意味着最多 3～4 次磁盘 I/O 就能找到数据，远快于全表扫描。

叶子节点用双向链表串联范围查询（>、<、between、like）可以顺序读取，磁盘效率极高，还能触发预读。

索引快是因为减少磁盘 I/O；MySQL 用 B+ 树，矮胖层数少、I/O 极少；叶子节点有序链表，范围查询极快。
