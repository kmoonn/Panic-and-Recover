# 慢查询优化

```text
开启慢查询日志，抓取慢 SQL
explain 分析执行计划，看 type、key、rows、Extra
解决全表扫描、索引失效问题
优化索引设计（联合索引、最左匹配、覆盖索引）
改写 SQL：避免函数、大分页、不合理联表
架构优化：缓存、分库分表、读写分离
```

- 定位慢查询

开启慢查询日志：执行时间超过2s、记录没走索引的sql

- 先执行explain

1. type访问类型（优先级）

system > const > eq_ref > ref > range > index > ALL

绝对禁止：ALL 全表扫描

2. key 实际用到的索引

为空 = 没走索引

3. rows 扫描行数

越小越好

4. extra

- 没加索引 / 索引失效

索引失效场景：字段函数操作、隐式类型转换、like %xx 左模糊、or 两边字段都未建索引、联合索引不满足最左匹配原则

- 分页深度过大

偏移量超大，需要扫描前面巨量行再丢弃

延迟关联 / 主键分页

- select * 冗余字段

回表查询增多、网络传输大、无法走覆盖索引

- order by group by 无索引

排序、分组字段加入联合索引末尾

- 多表联查不合理
- 事务、锁、连接等待
