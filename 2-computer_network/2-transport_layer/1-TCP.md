# TCP 传输控制协议

一种面向连接的、可靠的传输层协议。

可靠：按序交付、差错检测、丢包重传、流量控制、拥塞控制。

## 三次握手

- 第一次握手 SYN
  - 客户端 → 服务端：发送 **SYN 包**，随机初始序列号 **seq = x**
  - 客户端状态：**CLOSED → SYN_SENT**
- 第二次握手 SYN + ACK
  - 服务端 → 客户端：回复 **SYN+ACK 包**
    - 自己的随机序列号 seq = y
    - 确认号 **ack = x + 1**
  - 服务端状态：**LISTEN → SYN\_RCVD**
- 第三次握手 ACK
  - 客户端 → 服务端：发送 **ACK 包**
    - 序列号 **seq = x + 1**
    - 确认号 **ack = y + 1**
  - 双方状态：都变为 **ESTABLISHED**
  - 连接建立成功

**一握 SYN 我要连，二握 SYN+ACK 同意连，三握 ACK 连上了，双方 ESTABLISHED。**

三次握手是为了保证通信双方的「发送能力」和「接收能力」都正常，并且同步双方初始序列号 ISN（为可靠传输做准备）。

### 两次握手能行吗

如果只使用两次握手，服务端在发送 SYN+ACK 后就认为连接建立，但无法确认客户端是否收到。

若网络中存在延迟的旧 SYN 包，服务端会建立无效连接（与客户端丢弃的SYN建立连接），浪费资源，甚至导致 SYN 洪水攻击。

### 第三次握手能传数据吗

可以，TCP 第三次握手的 ACK 报文段中，可以携带应用数据，因为此时客户端已认为连接建立成功，这是 RFC 标准允许的优化。

第三次握手的包是 ACK 包，客户端在发送第三次握手的 ACK 报文段时，可以在同一个报文段中携带应用层数据。

因为此时客户端的状态已经是 ESTABLISHED，客户端知道连接已经建立成功，所以可以直接把数据塞进这个 ACK 包一起发过去。

服务端收到这个 ACK 后：先确认连接，进入 ESTABLISHED，然后再处理包里携带的数据。

减少 1 个 RTT 的延迟，不用等连接建立完再发数据，而是握手 + 发数据一起完成。

## 四次挥手

第一次挥手（FIN）
主动关闭方 → 被动关闭方：发送 FIN 包，seq = u
主动方状态：FIN-WAIT-1

第二次挥手（ACK）
被动关闭方 → 主动关闭方：回复 ACK 包，ack = u + 1
被动方状态：CLOSE-WAIT
主动方状态：FIN-WAIT-2
进入半关闭状态：主动方不再发数据，但被动方可继续发

第三次挥手（FIN）
被动关闭方 → 主动关闭方：发送 FIN 包，seq = v
被动方状态：LAST-ACK

第四次挥手（ACK）
主动关闭方 → 被动关闭方：回复 ACK 包，ack = v + 1
主动方状态：TIME-WAIT，等待 2MSL 后进入 CLOSED
被动方收到 ACK 后：直接进入 CLOSED

**一握 FIN 我要关，
二握 ACK 收到了，
三握 FIN 我也关，
四握 ACK 全关完，
TIME_WAIT 等 2MSL。**

## 发送窗口大小

TCP 发送窗口大小由**流量控制**的**接收窗口** rwnd 和**拥塞控制**的**拥塞窗口** cwnd 共同决定，**取两者最小值**。

rwnd 由接收方根据**缓存大小**动态反馈；

cwnd 由发送方通过**慢开始、拥塞避免、快重传、快恢复算法**自适应调整。

发送窗口控制发送速率，既不撑爆接收方，也不拥塞网络。

## 粘包拆包问题

TCP 是字节流协议，无消息边界，所以会产生粘包拆包。
解决方法有三种：固定长度、分隔符、消息头 + 消息体。
企业最常用的是消息头携带长度的方式，接收端根据长度精确读取消息，解决边界问题。

TCP 流式传输无边界，
粘包拆包很常见。
固定长度太浪费，
分隔符要转义。
最稳方案：
头带长度，按长读体！

## 可靠传输五大核心机制保证

三次握手建立连接；
确认应答 ACK 机制；
超时重传机制；
滑动窗口实现流量控制；
拥塞控制防止网络崩溃。
配合校验和、去重、排序等机制，最终实现数据无差错、不丢失、不重复、按序到达的可靠传输。

超时重传
滑动窗口（流量控制）
拥塞控制（慢启动、拥塞避免等）
校验和校验数据完整性
